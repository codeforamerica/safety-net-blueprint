# Application / Verification — Contract Overview

> Generated from source YAML files. Do not edit this file directly — changes will be overwritten on the next export.

This document describes the complete behavioral contract for the **Application / Verification** resources. It is intended for product owners, policy staff, and other non-technical reviewers.

## How to read this document

- **States** — the lifecycle stages each resource can be in, and how each affects SLA tracking.
- **Transitions** — actor- or system-triggered actions that can move a resource to a new state or update it in place.
- **Events** — automatic reactions to named domain events, including object creation, field changes, and timer callbacks.
- **Guards** — named conditions that control who can trigger a transition. If a guard fails, the request is rejected.
- **Procedures** — named step sequences called from transitions and events to encapsulate reusable logic.

To propose a change, there are two paths:
- **Non-technical:** Edit the CSV files in this folder and ask a developer to run `npm run contract-tables:import` to apply your changes to the source YAML.
- **Technical:** Edit the source YAML files directly and submit a pull request.

## States

The **SLA clock** tracks time toward resolution:
- **Running** — time is counting toward the SLA deadline
- **Paused** — time is not counting (resource is blocked, waiting on external input)
- **Stopped** — work is complete; SLA is no longer tracked

## Procedures

Named step sequences called from transitions and events. They encapsulate reusable logic.

| ID | Scope | Description | Parameters |
|----|-------|-------------|------------|
| `openApplication` | Application | Transition the application from submitted to under_review when a caseworker claims the intake review task | — |
| `createInterview` | Application | Create an Interview record when a caseworker claims an application_review task; SNAP requires an interview before determination (7 CFR § 273.2(e)) | — |
| `createProgramVerifications` | Application | Create electronic Verifications per member and document Verifications at the household level for the given program | program, memberCategories, householdCategories |
| `updateVerificationOnCallResult` | Application | Transition the Verification based on the service call result; on inconclusive, creates a document fallback per ex parte rules (42 CFR § 435.911) | — |
| `linkAppointment` | Application | Append the appointmentId to Interview.appointments when an appointment is scheduled against an interview subject. Non-interview appointments (subjectType != interview) are ignored. | — |
| `satisfyVerificationOnDocumentUpload` | Application | Satisfy the Verification and record the uploaded document version as evidence; trigger only fires when metadata.intake.verificationId resolves to a known Verification | — |
| `recordDetermination` | Application | Write eligibility outcome to ApplicationMember.programDeterminations. Informational write-back only — does not trigger application close. Medicaid RTE results may arrive before intake closes; SNAP results typically arrive after. | — |
| `initiateServiceCalls` | Verification | Route each electronic Verification to its required service calls by category; document-type Verifications are skipped by initiateServiceCall | — |
| `initiateServiceCall` | Verification | Post a single service call for this Verification; no-ops for document-type Verifications | serviceId |
