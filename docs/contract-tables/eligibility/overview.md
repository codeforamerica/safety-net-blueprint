# Determination Workflow — Contract Overview

> Generated from source YAML files. Do not edit this file directly — changes will be overwritten on the next export.

This document describes the complete behavioral contract for the **Determination** resource. It is intended for product owners, policy staff, and other non-technical reviewers who need to understand or propose changes to task lifecycle behavior.

## How to read this document

- **States** — the lifecycle stages a Determination can be in, and how each affects SLA tracking.
- **Transitions** — the actions that move a task from one state to another. Actor-triggered transitions are called explicitly by a person or system; timer-triggered transitions fire automatically after a set amount of time. Each transition lists who can trigger it (via guards), and what happens when it fires (effects).
- **Guards** — named conditions that control who can perform a transition. If a guard fails, the transition is rejected.
- **Request bodies** — the data a caller must (or may) include when triggering a transition.
- **Rules** — automated logic that runs at key moments (task creation, field updates, certain transitions) to assign tasks to queues and set their priority. Rules are evaluated in order; the first matching rule wins.

To propose a change — for example, adding a new state, changing who can escalate a task, or adjusting a routing rule — there are two paths:
- **Non-technical:** Edit the CSV files in this folder and ask a developer to run `npm run contract-tables:import` to apply your changes to the source YAML.
- **Technical:** Edit the source YAML files directly and submit a pull request.

## States

Tasks move through a defined set of states. The **SLA clock** tracks time toward resolution:
- **Running** — time is counting toward the SLA deadline
- **Paused** — time is not counting (task is blocked, waiting on external input)
- **Stopped** — work is complete; SLA is no longer tracked

| State | SLA Clock |
|-------|-----------|
| pending | Running |
| in_progress | Running |
| completed | Stopped |
| withdrawn | Stopped |

## Transitions

### Actor-triggered

These transitions fire when a caseworker, supervisor, or the system calls the corresponding endpoint (`POST /tasks/{id}/{trigger}`).

| Trigger | From | To | Guards | Effects |
|---------|------|----|--------|---------|
| `flag-expedited` | pending |  | callerIsSystem | Set `expeditedFlagged` → `true`<br>Emit `expedited` event |
| `begin-evaluation` | pending | in_progress | callerIsSystem | Re-evaluate undefined rules |
| `evaluate-final` | in_progress |  | callerIsSystem | Re-evaluate undefined rules |
| `complete` | in_progress | completed | callerIsSystem | Set `completedAt` → current time<br>Emit `determination_completed` event |
| `withdraw` | pending, in_progress | withdrawn | callerIsSystem | Set `withdrawnAt` → current time |

### Timer-triggered

These transitions fire automatically based on elapsed time — no actor action is required.

## Guards

Guards are conditions checked before a transition fires. A transition will not execute unless all of its guards pass. Multiple guards on a transition use AND logic; `any of:` within a guard uses OR logic.

| Guard | Condition |
|-------|-----------|
| `callerIsSystem` | `$caller.roles` contains any of `system` |
